Java安全–sql注入

Java中操作sql语句的方法主要有几种
java.sql.Statement
java.sql.PrepareStatement
使用第三方 ORM 框架 —— MyBatis 或 Hibernate

先说第一种

java.sql.Statement

java.sql.Statement跟php连接数据库的方式很像，就是实例化一个String类来作为sql查询语句带进后端数据库查
类似这种

这种没啥好说的，该注入注入吧，没救了

java.sql.PrepareStatement

第二种是java.sql.Statement的进阶版接口，他会对传参进行预编译，也就是java.sql.PrepareStatement

它使用了sql语句中的占位符 ？ ，代替需要传入的参数，然后在传给数据库
大概是这样写的

我理解的流程是首先设置需要传入sql语句中的内容为（preparedStatement.setString(1,reqStuId);）然后将要带入查询的占位符的内容进行编译，比如传入的参数是 ‘ or 1 # ，那么编译后的结果就是 ‘\ or 1 # ,’ 就无法起到闭合的作用，再加上占位符的内容需要自带带引号包裹，所以最后带入查询sql语句中的样子为

SELECT * FROM user WHERE id = ''\ or 1 #'

就什么锤子都不能用了

order by绕过

但预编译会出现一种特殊情况导致绕过，即order by；比如

select * from user order by id

那这种情况下加入用预编译的手法就算是正常的传入参数也会变成 select * from user order by ‘id’ ；即把’id’放进数据库中查，这必然是查不出东西的;

解决-硬编码

所以当传递的参数为字符串类型时会考虑使用硬编码的方式，即直接在代码中写死了order by后面的id值

    String column = "id";
    switch(column){
        case "id":
            sql="select * from user order by id";
            break;
}

当id定义为int类型（整型，不能加’）时就不会产生sql注入，因为java时强类型语言，前后端类型不一致会报错

以上问题类似的还有group by等

Mybatis

mybatis执行流程

mybatis本身是apache的开源项目ibatis改名，是一个底层框架，对jdbc操作数据库的过程进行封装，让开发者主要精力在sql语句那边而不是类似创建连接等其他代码上

大概的框架主要是

  1. 提前配置好操作的sql语句

    SqlMapConfig.xml，此文件作为mybatis的全局配置文件，配置了mybatis的运行环境等信息。

    mapper.xml文件即sql映射文件，文件中配置了操作数据库的sql语句。此文件需要在SqlMapConfig.xml中加载。

  2. 建立会话

    通过mybatis环境等配置信息构造SqlSessionFactory即会话工厂;
    由会话工厂创建sqlSession即会话，操作数据库需要通过sqlSession进行;
    mybatis底层自定义了Executor执行器接口操作数据库，Executor接口有两个实现，一个是基本执行器、一个是缓存执行器;

** Mapped Statement也是mybatis一个底层封装对象，它包装了mybatis配置信息及sql映射信息等。mapper.xml文件中一个sql对应一个Mapped Statement对象，sql的id即是Mapped statement的id。**

  1. jdbs的解析处理

    Mapped Statement对sql执行输入参数进行定义，包括HashMap、基本类型、pojo，Executor通过Mapped Statement在执行sql前将输入的java对象映射至sql中，输入参数映射就是jdbc编程中对preparedStatement设置参数。

    Mapped Statement对sql执行输出结果进行定义，包括HashMap、基本类型、pojo，Executor通过Mapped Statement在执行sql后将输出结果映射至java对象中，输出结果映射过程相当于jdbc编程中对结果的解析处理过程。

总结起来mybatis就是七个步骤

  1. 加载核心配置文件SqlMapConfig.xml
  2. 创建SqlSessionFactoryBuilder对象
  3. 创建SqlSessionFactory对象（2,3可以写一起）
  4. 创建SqlSession对象
  5. 执行SqlSession对象执行查询，获取结果User
  6. 打印结果
  7. 释放资源

mybatis中的预编译手法-内联参数#{}

在mybatis中有两种内联参数（就当参数看吧）
其中第一个#{}:一个占位符。preparedStatement向占位符中设置值，自动进行java类型和jdbc类型转换。#{}可以有效防止sql注入。

第二个${}:表示拼接sql串，通过${}可以将parameterType 传入的内容拼接在sql中且不进行jdbc类型转换。

简单对应着jdbc中有预编译跟没预编译的两种手法，自然${}跟java.sql.Statement一样会产生注入的风险，#{}的底层就是java.sql.PrepareStatement那一套，所以order by那套问题也存在（order by的问题用上面的硬编码方式解决）

这里面#{V} 告诉mybatis创建一个预处理语句参数，在JDBC中，这个的一个参数在Sql中会由一个“ ？”表示，并被传递到新的预处理语句中

所以上面test类用jdbc的写法长这样

String findUserById ="select * from user where id = ?";
PreparedStatement ps = conn.prepareStatement(findUserById);
ps.setInt(1,id);

也就是预编译的java.sql.PrepareStatement写法

namespace:用来区别不同的类的名字

id:标识映射文件中的sql，称为statement的id将sql语句封装到mappedStatement对象中，所以将id称为statement的id

parameterType:指定输入参数的类型.

resultType:指定输出结果类型。mybatis将sql查询结果的一行记录数据映射为resultType指定类型的对象。如果有多条数据，则分别进行映射，并把对象放到容器List中

模糊查询like的兼容导致注入

另外还多了一个问题，即模糊查询LIKE
LIKE操作符用于在WHERE子句中搜索列中的指定模式
常见的几种搜索方式

一般用like语句的时候都是用${}的内联参数，因为如果用#{}的话，sql的xml中就是这样写的

select * from user where username like '#${value}%'

转换成实际的操作语句就会变成

select * from user where username like ''value''

(左右都是两个单引号，带进数据库里查询肯定报错)

concat函数防止存在like时可能产生的注入

为了使like函数能够执行，只能用${}的内联参数，那么就要考虑如何防止sql注入，这时候就需要用到sql原生的函数concat了

concat函数的目的是将多个字符串合并，例：select concat（’A’,’B’）; 结果为AB

那么防注入的demo如下

当传入值为恶意的1’ or 1#时，经过#{}内联函数时会变成 ‘1’\ or 1 #’ ，最终的sql语句为

select * from student where student.stu_name like '%1'\ or 1 #%'

成功拦截了注入
